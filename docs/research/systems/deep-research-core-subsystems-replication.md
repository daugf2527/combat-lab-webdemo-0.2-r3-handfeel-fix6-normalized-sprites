# DNF/DFO 三个核心子系统的 1:1 实现研究报告

## 版本基准

如果你的目标是“当前 live 版本的 1:1”，第一件事不是写代码，而是先锁死“服务器 + 年代 + 赛季”。这几个系统在近两年并不是小修小补，而是实打实地被重做过：装备继承/转换规则在 115 级赛季改过，技能系统里的 TP 在韩服与中服都被删除并并入基础技能，疲劳值在不同服务器也不是一个常数。因此，开发上必须把规则按 `server_id / season_id / patch_id` 做版本化，而不能写成“全服通用一套逻辑”。如果你的目标是当前 live 版本，应该把 entity["company","Neople","game developer"] / entity["company","Nexon","game publisher"] / entity["company","Tencent Games","game publisher"] 的公开更新说明当成第一证据层，而不是把老端、旧赛季资料、甚至私服资料混进来。citeturn19view0turn21view1turn27search4turn26view2

本报告采用的“可公开验证基准”是：韩服/DFO 115 级赛季后的公开指南与更新说明，外加韩服官方社区上能复核的玩家实测与界面记录；**不把泄露客户端、私服表、来历不明的逆向包当主证据**。原因很简单：你现在要做的是“当前规则 1:1”，而不是“把旧端里残留的系统拼贴起来”。一个直接例子就是 TP——当前公开规则里它已经不是玩家可分配的树了，但很多旧资料仍然把它当核心系统。citeturn21view1turn27search4turn26view3

先给一个开发结论：如果你要做“当前 DNF/DFO 的骨架”，三层必须分开建模。第一层是**装备成长状态**，包括强化/增幅/锻造/附魔/继承；第二层是**技能配置状态**，包括 SP、内建 TP 效果、VP（Skill Evolution）和 Skill Enhancement；第三层是**副本规则状态**，包括疲劳消耗、难度档位、队伍人数缩放策略。三层之间通过角色面板读写，但规则表不能互相硬编码。citeturn38view0turn21view0turn32search1turn19view1

## 装备成长系统

### 强化

当前公开可验证的强化成功率曲线，可以拼出完整的 `+0 → +15`：`0→1/1→2/2→3/3→4 = 100%`，`4→5 = 80%`，`5→6 = 70%`，`6→7 = 60%`，`7→8 = 90%`，`8→9 = 80%`，`9→10 = 70%`，`10→11 = 40%`，`11→12 = 30%`，`12→13 = 8%`，`13→14 = 1%`，`14→15 = 0.5%`。这条曲线来自官方活动/商品页所展示的强化概率说明；而当前装备系统官方指南则明确了失败惩罚规则。citeturn6search0turn16view1

实现上最关键的不是概率，而是**失败分段**。当前官方指南写得很清楚：武器 `+0~+10` 失败无惩罚，`+10→+11` 与 `+11→+12` 失败都是 **-3 级**，`+12→+13` 及以上失败会**碎装**；如果持有装备保护券，则不会碎，但会**直接归零到 +0**。防具、首饰、特殊装备则是 `+0~+10` 失败无惩罚，`+10→+11` 起失败直接碎装，保护券同样是“保装备、不保强化等级”，结果仍是 **归零到 +0**。这一点对服务端很重要：保护券不是“防碎且保级”，而是“防碎但重置强化值”。citeturn16view1turn38view0

当前版本还有一套**安全强化**，它不是保护券的别名，而是独立系统。官方指南给出的逻辑是：安全强化只适用于 `100 级以上、独特及以上、+11 及以下的武器`，消耗的是 Leiern Core 而不是无色；失败不掉级，但会给下一次叠加保底修正。公开表里 `+4→+5` 到 `+9→+10` 失败修正是 `+5%p`，`+10→+11` 是 `+2%p`，`+11→+12` 是 `+1%p`；对应成功率分别是 `80/70/60/50/40/30/8/3`。所以你的实现不能只做一个 `is_protected` 布尔值，而要把“普通强化”和“安全强化”做成两套 rule set。citeturn38view0

另外，当前 DFO 公开指南还写明：武器与耳环在 `+12` 以上会额外给**最终伤害**，`+12 = 0.4%`，`+13 = 0.7%`，`+14 = 1.0%`，`+15 = 1.2%`，`+16+` 每级再加 `0.2%`。这意味着强化不是单纯“改面板攻击/防御”的系统，而是高段位时会跨进独立乘区。面板层最好拆成 `display_stat_bonus` 与 `final_damage_bonus` 两条。citeturn38view0

### 增幅

增幅的底层对象不是“普通装备”，而是**带异界气息并完成净化的装备**。净化后，装备会获得四种异次元属性之一：`STR / INT / VIT / SPI` 中的一个，而且是随机给。韩服官方社区当前指南还补了一条非常关键的工程细节：如果刷出的是错误属性，可以再用“变异的扭转书”把异次元属性改成想要的那一项。这说明在数据模型里，`dimensional_stat_type` 必须是装备实例字段，而不是隐藏在增幅等级里的推导值。citeturn38view0turn15view0

当前公开可见的普通增幅成功率，能拼出 `+0 → +15`：`0→1/1→2/2→3/3→4 = 100%`，`4→5 = 80%`，`5→6 = 70%`，`6→7 = 60%`，`7→8 = 70%`，`8→9 = 60%`，`9→10 = 50%`，`10→11 = 40%`，`11→12 = 30%`，`12→13 = 20%`，`13→14 = 20%`，`14→15 = 20%`。而当前官方指南给出的失败惩罚是：`+0~+7` 无惩罚，`+7→+8`、`+8→+9`、`+9→+10` 失败分别**-3 级**，`+10` 以上失败则**碎装**；如果用了增幅保护券，装备不碎，但增幅值重置。增幅保护券的语义因此应实现为 `prevent_destroy = true, reset_value_on_fail = true`。citeturn7view1turn9search2turn38view0

官方文本里还给了增幅的“高段收益”：`+10 = 最终伤害 0.2%`，`+11 = 0.4%`，`+12 = 0.7%`，`+13 = 1.0%`，`+14 = 1.2%`，`+15 = 1.4%`，`+16+` 每级再加 `0.2%`。也就是说，增幅在系统层等于“强化收益 + 异次元属性收益 + 高段最终伤害收益”。因此服务端不要把它实现成 `if amplified then reinforced=false` 这种互斥替代，而应视为一个**复合加成系统**。citeturn9search2turn38view0

这里有一个很容易踩坑的“版本差异”。官方英文指南目前还公开了一张 **Lv.105 Epic / Mythic** 的增幅属性表，到 `+13` 为止分别是：Epic `12,14,16,18,21,25,30,34,61,75,91,105,130,168`，Mythic `13,15,17,20,22,27,31,36,64,78,94,111,134,174`。但**不要把这张表直接拿去当 115 赛季的 1:1 数据**。韩服 2026 官方社区指南已经给出当前 115 赛季的实机锚点：`+8` 增幅武器显示额外属性约 `+65`，`+10` 增幅 115 特殊装备示例显示“强化等效属性 +31，增幅额外属性 +96”。这和 105 表的 `+8 = 61 / +10 = 91` 已经不一致，说明 115 赛季的增幅属性表至少部分重算过。开发上要把“105 表”和“115 表”分表管理，不能偷懒复用。citeturn38view0turn15view0

安全增幅同样是独立系统。当前官方公开表是：`0→1/1→2/2→3/3→4 = 100%`，`4→5 = 70%`，`5→6 = 60%`，`6→7 = 50%`，`7→8 = 50%`，`8→9 = 40%`，`9→10 = 30%`；`4→5/5→6/6→7` 的失败修正是 `+10%p`，`7→8/8→9/9→10` 的失败修正是 `+5%p`。可用范围是 `100 级以上、独特及以上、+9 及以下装备`，也就是最多安全做到 `+10`。citeturn10search3turn38view0

### 锻造

当前官方指南对锻造给出的硬规则是：锻造只作用于**武器**，提升的是**独立攻击力**；上限是 `+8`；失败只消耗材料，**不会掉级，也不会碎装**。这几条是无争议的基础规则。citeturn16view0turn38view0

公开网络上能稳定复核到的锻造成功率，是 2021 年韩服因概率公开而在游戏内显示的那组：`0→1 = 100%`，`1→2 = 50%`，`2→3 = 50%`，`3→4 = 30%`，`4→5 = 30%`，`5→6 = 15%`，`6→7 = 15%`，`7→8 = 7.5%`。这组数字在韩服社区、Ruliweb 与其他记录页是一致的，但**当前官方指南文本本身没有再把它逐级印出来**；因此它适合作为“公开可验证的高可信近似基线”，但如果你要做“当前 115 赛季绝对精确”的产品，最好还是再用 live 客户端 UI 做一次人工复核。citeturn13search0turn13search2turn16view0

当前 115 赛季最关键的公开锚点，是韩服官方社区 2026 指南给出的比较值：`+8 锻造 = +1102 独立攻击`，而同一时期的 `+12 强化 = +961 独立攻击`，`+13 强化 = +1126 独立攻击`。这意味着在当前版本里，**固定伤害职业的武器成长逻辑已经不再是“永远只看锻造”**：在 `+13` 强化之前，`+8` 锻造仍是最快的独立攻击成长点；到了 `+13` 强化，强化武器的独立攻击会反超 `+8` 锻造。你在战斗公式层必须允许“强化和锻造都能影响独立攻击”，但最终采用高值或按官方逻辑合成，而不是只让锻造独占独立攻击。citeturn15view0

目前公开文本**没有**给出完整的“115 赛季 `+1~+8` 锻造每级独立攻击增量表”。这一项是当前公开资料的真实空白。你可以确认规则、上限、失败行为与 `+8` 终点值，但如果你要把每一级都精确到数字，公开网页资料还不够。citeturn16view0turn15view0

### 附魔

附魔系统的底层规则非常清楚：**一件装备只能有一条附魔**，而且必须匹配卡片/宝珠允许的部位；给已经附魔过的装备再次附魔时，**旧附魔会被新附魔完全覆盖**，不会叠层，也不能回收旧附魔。开发上最安全的做法，是给每个装备实例只保留一个 `enchant_instance_id`，重附魔时直接替换，不要做附魔列表。citeturn18view1turn18view2

卡牌和宝珠并不是两套效果逻辑，而是两种交付形式。卡牌需要附魔师或他人的附魔商店，加上对应材料才能上装备；宝珠则是消耗品，可直接使用。官方指南还说明：并不是所有附魔都有卡片形态，有些只以宝珠形态产出；卡与宝珠在发生一次交易后会转成**账号绑定**。这意味着你的道具系统里最好拆出 `delivery_form = card|bead` 与 `tradability_state` 两个字段，而不要把“卡”和“宝珠”当成不同附魔体系。citeturn18view1

当前版本的附魔来源也已经不是“只有打怪掉卡”那么简单。官方当前 DFO 附魔指南按部位直接列出了 Monster Card/Bead 与 Merchant Guild Silver Coin Bead 两个来源层级；韩服官方指南还写明卡片可以升级，也可以同稀有度合成，有几率升一阶。系统层真正应该实现的是：**附魔效果来源可多样，但只要落到装备上，运算全都统一为单槽附魔**。公开规则里也没有看到专门针对附魔的“同名 unique group 排斥”机制，所以跨装备部位的附魔效果应按普通属性线性累加，把“唯一排斥”留给装备词条系统，而不是附魔系统。citeturn18view0turn18view1

### 继承

115 赛季以后的“继承”，不能再按早期“把强化值拷过去”去理解。官方更新说明和便利性改版都强调了一个重点：**继承/刻印会交换两件装备上的强化效果包**。这个“效果包”不是只含强化值，而是同时包含 `强化/增幅/锻造、附魔、融合石` 等增强效果。也就是说，继承实现上更像两个装备实例的 enhancement bundle swap，而不是单向复制。citeturn19view1turn19view2

当前 115 大版本公开写明：`110 级装备` 与 `115 级稀有/独特/传说/史诗/太初装备` 之间可以继承，但**可交易装备不能继承**。在 115 的批量继承里，官方还给出了成本：每个部位 `25,000 金币` 或 `恶意残渣 x5`。另外，便利性改版说明还强调：发生过继承/刻印之后，带强化效果的装备会变成**不可超越**。所以你需要至少做三个校验器：`same_part_check`、`tradable_forbidden_check` 和 `transcend_block_after_swap`。citeturn19view0turn19view1turn19view2

115 赛季还有一个很容易跟继承混淆的系统叫**装备转换**。它只作用于能注册进 Armory 的 115 装备，转换后会保留 `强化/增幅/锻造/融合石/装备品级/附魔` 等全部增强信息；如果调律同部位同稀有度，还能保留调律值。也就是说，当前 live 版本实际上并存了三种“装备成长迁移”路径：早期赛季的选项转移、115 的继承交换、115 Armory 装备转换。三者一定要分开做。citeturn19view1

## 技能树与技能进化

### 当前 live 真相：TP 已经不是玩家树了

如果你的目标是**当前韩服/DFO/中服的 115 赛季**，那就不要再实现“老 TP 树”。韩服官方 115 赛季说明明确写了：**TP 系统删除**，原本 TP 提供的强化效果，会在“尽量保持角色性能不下降”的前提下，按 TP 满级效果迁移到各技能的基础规格里；部分会造成手感倒退或功能过强的 TP 选项不会原样迁移，而是调整后才并入。中服 2025-01-15 的官方职业技能优化页也同步写了“删除特性点系统”，大量技能改动说明里直接标注“删除特性点系统”或“TP 消耗量变更后删除”。这不是传闻，而是公开 patch note。citeturn27search4turn26view2turn26view3

这件事对开发的含义非常大：**当前版本的技能数据表，应该存的是“最终 baked 后的基础技能参数”，而不是“基础技能 + 可购买 TP 修正”**。也就是说，范围增加、攻击速度增加、命中次数调整、额外 hit、CD 调整等，以前挂在 TP 上的东西，在 current live 里大多数已经被内建到 `skill_base_spec` 里了。你的技能编辑器如果还保留老 TP UI，只会把自己带偏。citeturn27search4turn21view1

### SP 系统目前公开能确认到什么

官方当前技能窗口说明只明确了“Learn Skill 页签使用 SP 点学习技能”，但**没有公开一张完整的“当前版本每级获得多少 SP”的总表**。这意味着如果你想凭公开网页直接抄出当前满级总 SP，你会碰到证据空白。官方真正公开出来的是：SP 仍是学习页的基础货币；真觉醒带来的旧“部分 SP 消耗减免系统”已经被删除并并入技能基础规格；二觉主动在当前版本会变成 `0 SP` 且自动习得。citeturn21view1turn26view3

但公开资料仍然给了几个很重要的“当前 SP 锚点”。中服 2025 技能优化文档里，可以直接看到：一些基础技能如 `1 级技能` 的 `技能点消耗量 = 20`；一觉被动从 `45 → 30`；二觉被动从 `80 → 45`；二觉主动从 `200/100 → 0` 并改成自动习得。韩服更新说明里，70 级新主动技能的公开范例则写明 `SP 消耗量 = 70`，`Master/Max = 40/50`，而其原 TP 强化技能为 `TP 消耗量 = 2，Master/Max = 5/7`。把这些点连起来，你会发现：**当前技能树已经不适合由“技能学习区间 → 自动推出 SP/TP 成本”来驱动，而必须逐技能存储 `sp_cost_per_rank`、`master_level`、`max_level`、`auto_learned`**。citeturn26view2turn26view3turn22search9

也就是说，用户问的“各技能 SP 消耗梯度”在当前版本里，不应该被你实现成一张简单的“15 级都是 20 SP、20 级都是 25 SP”这种统一表。因为公开 patch note 已经反复证明：职业重做会把单个技能从 `20` 改到 `50`，会把被动从 `40/50` 改成 `1/11`，会把二觉主动直接改为自动习得。**正确的 1:1 做法是逐技能存成本，而不是按等级段推成本。**citeturn28search0turn28search4turn26view2

### Skill Evolution 与 Skill Enhancement

当前版本真正替代旧 护石/符文 体系的，是 **Skill Evolution (VP)** 与 **Skill Enhancement**。官方 DFO 指南和更新页写得非常直接：VP 在真觉醒/自觉醒后解锁；总共有 `5` 点 VP；每个可进化技能提供 `2` 个选项，但同一技能一次只能选 `1` 个；通常覆盖 `35~80` 级主动技能、部分 `35~80` 被动，以及部分承接旧护石功能的主动/辅助技能。更关键的是，官方明确写明：**Skill Evolution 不改变“按冷却折算后的总攻击量”**，它改的是技能形态、释放方式、判定、追踪、聚怪、前后摇等；少量选项会为施放时长与攻击时长做平衡。citeturn21view0turn21view1

这等于给了你一个非常明确的技能建模方式：VP 不是“再加一个乘区”，而是一个 `behavior patch`。实现时，建议把 VP 做成对基础技能脚本的**分支补丁**，它可以改 hit list、射程、追踪、前摇、后摇、位移轨迹、额外对象、关联技能触发，但不要把它当一个单纯数值 buff。因为官方已经讲明，VP 的设计目标不是平白加总伤，而是改“技能形态”。citeturn21view0turn21view1

与 VP 并行的是 **Skill Enhancement**。它不是老 TP 的复刻，而是新的三选技能强化层。官方规则是：最多选 `3` 个技能；每个技能只能从两种方案中选一个。对 DPS 来说，`15~30` 级主动技能的两种方案是 `技能攻击力 +60%`，或 `技能攻击力 +43% + 冷却 -15%`；`35~80` 级主动技能则是 `+55%`，或 `+38% + 冷却 -15%`。对 Buffer，则在这些方案上额外附带 `主属性 +40`，治疗技能还有“治疗量增加 / 治疗量小增 + 冷却减少”的变种。这个系统才是当前版本替代老 TP“伤害/冷却细修”的那个层。citeturn21view0turn21view1

因此，如果你要做 current live 的 1:1 技能系统，正确的层次应该是：`基础技能 (含 baked TP)` → `SP 学习等级` → `VP 技能形态分支` → `Skill Enhancement 三技能数值分支`，而不是 `基础技能 → TP 树 → 护石 → 符文`。后者已经不是现行结构了。citeturn27search4turn21view1

## 疲劳与副本难度

### 疲劳值

如果你按“156/176 + NeoPremium”去写 current DFO Global，你会直接写错。DFO Global 官方基础指南和 Neo Premium 页面当前都写得很清楚：**基础疲劳值是 156，Neo Premium Contract 作用后是 273**，不是 176。韩服官方指南则写的是 `156` 基础疲劳，外加 `PC 房额外 78`，而且**额外疲劳优先消耗**。所以“156/176”更像旧资料残留，不适合再做 current live 的硬编码。正确做法是把疲劳基数按 `server_id + benefit_profile_id` 参数化。citeturn34search1turn33search1turn34search2turn34search3turn32search1

普通地下城的疲劳规则，韩服官方指南给的是最朴素那套：**每进一个房间消耗 1 点疲劳**；只要还有 `1 FP` 就能进图；一旦已经在图里，即使疲劳清零，依然可以继续移动到后续房间。官方还特别提示“有些地下城会随难度提高房间疲劳消耗”。也就是说，疲劳消耗逻辑不能做成“进图先扣固定值”这一种。你至少要支持三种 policy：`per_room`、`fixed_on_entry`、`special_override_by_difficulty`。citeturn32search1turn34search1

当前赛季的特殊图里，固定入场疲劳依然大量存在。韩服社区当前资料明确写到：`종말의 숭배자 / Disciple of Doom` 每次入场固定消耗 `8 FP`；某些任务/任务型地下城会出现 `房间 2 FP`，而且同样允许在只剩 `1 FP` 时进入；115 赛季多种上级地下城的官方说明则写成**入场固定 30 FP**。另外，韩服 2026-03-11 的官方更新说明又把新赛季剧情地下城改成了**不消耗疲劳**。所以疲劳在 current live 里已经是“按内容类型配规则”，不是一套游戏全局常量。citeturn32search4turn32search2turn35search8turn31search0

### 难度梯度与队伍人数缩放

公开官方指南对“难度”的表达，更多是**名望门槛与可选档位**，而不是把 HP/攻击倍率印成公式。举例来说，110 级一般地下城 `Canyon Hill` 的公开条目列出了 `Normal / Expert / Master / King / Slayer` 五档，对应的名望门槛分别是 `4,176 / 13,195 / 23,259 / 29,369 / 33,989`。这说明在设计上，难度档是每个 dungeon profile 自己携带的，而不是一个全局固定枚举之后只改倍率。citeturn32search10turn37search11

但用户最关心的“Normal→Expert→Master→King 的**实际 HP/伤害倍率**”，公开官方网页文本**没有给出一张可直接抄表的通用倍率表**。目前公开世界里最接近这类信息的是韩服官方社区玩家维护的“110 级地下城体力/状态抗性表”，它确实在持续更新，但主要数值嵌在图片里，网页文本可检索部分只确认了这是一个长期维护的实测表，而没有把完整倍率拆成纯文本。对 1:1 开发来说，这意味着你不能负责任地声称“官方公开了通用 N/E/M/K 倍率表”；它并没有。citeturn36view0

队伍人数缩放同样**不是一条全局公式**。当前公开资料至少能确认三类策略并存。第一类是**显式 1~4 人预设**：韩服玩家在官方社区讨论“白海上级地下城”时明确提到，某些内容的 UI 会直接显示按 `1/2/3/4 人` 分开的队伍血量规则。第二类是**2 人以上直接按 4 人盘子算**：当前韩服社区多个问答都写到，一些 endgame 队伍本在 `2/3/4 人` 时会直接按 `4 人 HP` 处理。第三类是**特殊系统强制 4 人难度**：韩服 2026 官方更新说明里，“上级地下城雇佣 Buffer 的佣兵团支援队伍”即使只有最多 3 名真人，也会强制应用 `4 人 기준` 的地下城难度。citeturn37search18turn37search1turn37search2turn35search0

所以，真正应该落地到代码里的不是一个 `party_hp_multiplier[1..4]` 全局数组，而是**每个副本自己声明 party scaling policy**。至少要支持 `solo_scaled`、`explicit_1to4_scale`、`fixed_4p_when_party`、`forced_4p_support_party` 这几种。你如果把全游戏都写成“1/2/3/4 线性乘血”，后面一定会和 current live 对不上。citeturn35search0turn37search7turn37search18

## 适合开发落地的数据结构

结合上面的公开规则，当前版本最稳妥的实现方式，是把状态和规则表彻底拆开。装备侧至少要有 `equipment_instance`、`enhancement_state`、`amplification_state`、`refinement_state`、`enchant_state`、`inheritance_rule_set`、`conversion_rule_set` 七个层级。这样你才能正确支持“强化/增幅互斥但都是 enhancement bundle 的一部分”“继承时交换整包增强效果”“Armory 转换时保留增幅/锻造/附魔”等当前 live 规则。citeturn19view1turn19view2turn38view0

技能侧建议最少拆成 `skill_base_spec`、`skill_rank_spec`、`skill_sp_rule`、`skill_vp_option`、`skill_enhance_option` 五层。`skill_base_spec` 里就应该直接包含 baked TP 后的基础行为，因为当前版本 TP 已经不再是独立玩家树；`skill_sp_rule` 里要逐技能存 `learn_level`、`sp_cost_per_rank`、`master_level`、`max_level`、`cube_cost`、`auto_learned`；`skill_vp_option` 必须能覆盖行为补丁；`skill_enhance_option` 则负责三技能数值分支。这样你才能既兼容当前 VP，又不被老 TP 思维绑架。citeturn27search4turn21view0turn21view1

副本侧则应至少拆出 `dungeon_fp_rule`、`difficulty_profile`、`party_scaling_profile`。`dungeon_fp_rule` 负责 `per_room / fixed_entry / no_fp / difficulty_override`；`difficulty_profile` 不只存名字，还要存门槛、AI 套件、房间结构和可能的单独倍率；`party_scaling_profile` 则必须是每图自带，而不是全局共享。只有这样，你才能同时表示“一般图 1FP/房间”“特殊图固定 8FP”“剧情图 0FP”“同一 dungeon 有 1~4 人预设或 2+ 直接按 4 人盘子”的差异。citeturn32search1turn32search4turn31search0turn35search0turn37search18

## 已确认与未确认项

已经可以公开确认、足以直接指导开发的内容包括：强化 `+0→+15` 成功率曲线、强化分段惩罚与保护券语义、增幅的异次元属性随机化方向与安全增幅规则、当前版本“TP 删除并并入基础技能”的事实、VP 与 Skill Enhancement 的精确点数/档位规则、附魔单槽覆盖规则、115 赛季继承/转换的边界与保留项、以及当前 DFO/KR 下疲劳的基础框架。换句话说，**你完全可以先把系统骨架搭出来，而且能做到和 current live 的公开规则一致**。citeturn16view1turn38view0turn21view1turn18view1turn19view1turn34search1
