# DNF/DFO 战斗系统一比一复刻研究报告

## 最核心的结论

先给结论：你现在要做的不是“复刻一个抽象的 DNF”，而是必须先冻结成一个**区服 + 补丁日期 + 规则分支**。因为近两年中韩英服务已经出现了明确分叉：中文服在 2025 年 6 月 12 日上线“技能定制/技能突破”并移除了原有护石/符文系统；全球服在 2025 年 10 月 28 日上线 VP（Skill Evolution / Skill Enhancement）系统，同样明确删掉 Rune/Talisman；韩服在 2026 年的官方指南中还追加了技能连招、制压与 홀드链等新规则。如果你不先把目标指定成“国服 2025-06-12 之后”、或“韩服 2026-04-23 之后”、或“DFO Global 2025-10-28 之后”的某一个冻结版本，就不可能做出真正的一比一。citeturn16view1turn18view0turn18view1turn42view2turn30view0

第二个结论是：**公开官方资料已经足够还原“玩家可见层”的大量真值**，包括成长节点、SP/TP/VP、Buff 换装、强化/增幅/锻造/继承、套装点数、队伍/攻坚 UI、若干副本限制、制压/无力化/点火/反击等；但**不足以直接给出**命中/回避公式、随机种子、同 Tick 冲突裁决、技能帧窗、服务端归属与反作弊审计等“隐藏实现层”。所以你的研发路线应该是：**官方资料定行为，公开工具链定数据容器，线上实测/录制/抓包定黑箱细节**。citeturn33view0turn30view0turn40view5turn40view0turn40view3

下面这份报告**不使用泄露客户端、被盗源码或私服代码作为真值源**。我只把公开可见的官方中/英/韩资料、公开开源工具、公开社区材料作为证据链；其中开源逆向/私服后台只拿来判断“文件容器、表结构、后台依赖关系”，不拿来当官方行为真值。

## 战斗底层状态机

公开官方资料可以较高置信复原当前版本的“玩家可见战斗状态机”。首先是**无力化/点火/反击/无敌/超级护甲**：韩服官方指南明确写出，攻击会按伤害比例削减怪物的无力化槽；无力化槽清空后怪物进入短时间 groggy 并承受更高伤害。点火值随着你首次命中后持续上升，点火越高，无力化削减率越高；被怪物命中会使点火下降并暂时中断。反击状态不再覆盖怪物整个攻击过程，而只在某些特定时点进入；玩家的无敌与超级护甲也都有明确的视觉标记。citeturn33view0turn33view1turn33view2

其次是你最需要的一块：**制压与 Hold Chain**。韩服 2026 官方已经把它写成单独系统：部分怪物在特定机制阶段会出现“制压 UI”，带“制压力”的技能命中后会减少制压计数；处于制压 UI 时，相关技能的 Quick Slot 会被高亮。对不可抓取的敌人，原本带 Hold 效果的技能会被赋予制压力，而且官方明确说**制压力与 Hold 时间成比例**。更关键的是，官方还给出了组队时 groggy 延长的计算规则：技能导致的僵直延长，会把制压力按**除以队伍人数后再做小数向上处理**的方式应用；这已经足够你把 “单人/多人制压收益衰减”落成一个明确的服务端公式入口。训练房里还能召唤“制压用沙袋”做验证。citeturn30view0turn31view0

**目标标签与可命中对象层**，公开资料没有把内部枚举表直接放出来，但已经暴露出若干关键标签：`Named/Boss`、`Counter`、`Weakness`、`Immobility`、`Abnormal`、`Grab-immune`、`Hold-immune` 这些状态都是真实存在的判定标签，而不是纯 UI 文案。官方对 Neutralize 的说明还指出，某些 Named/Boss 具有不同弱点和不同槽量，槽破后会暴露给异常、控制、无力化等更多减益。也就是说，你的服务端一定不能只做一个“HP + SuperArmor”的简化状态机，至少要有**目标类型、控制免疫类别、弱点类别、阶段标签、反击窗口**这几层。citeturn33view1turn33view0turn21view1

**命中率 / 回避率 / Miss** 这一块，公开资料能确认“有这个系统”，但不能确认“精确公式”。官方礼包、光环、徽章、套装与装备词条都出现了 `Hit Rate`、`Evasion`、`Physical/Magical Critical` 这类显式战斗属性，说明命中与回避不是 UI 装饰，而是参与战斗的真实参数；但我没有在官方公开资料里找到当前版本命中/回避/Miss 的完整判定公式。因此你若要做一比一，只能通过**冻结版本后的录像比对 + 批量测样 + 回归拟合**去反推。citeturn27view2turn34search11turn23view2

**技能锁定、技能栏、快捷键与命令输入**层，当前公开资料已经相当具体。官方指南说明技能可以拖放到 quick slot；技能窗口可直接批量注册/取消冷却提示，也可以批量锁定/解锁命令键；新的技能导入/自动加点结果里还可以一并带上 `Command Lock`、`Quick Slot`、`Cooldown Alert` 这些附加信息。韩服还公开了 `Skill Chain`：二觉解锁、最多绑定 4 个技能、链槽会按“当前可施放”的顺序依次施放，带再输入功能的技能必须把再输入流程跑完才会轮到下一个技能。也就是说，技能系统不只是“学了什么技能”，而是至少要保存**学习状态、快捷栏映射、指令串、锁定态、冷却提示态、链槽态**五个子层。citeturn21view7turn21view6turn21view5turn18view0turn7search3

## 成长、SP/TP 与技能变体系统

成长分层现在可以做得很准。韩服和全球服官方指南都确认了：创建角色时就选择转职；一觉在 50 级、二觉在 75 级、三觉/Neo Awakening 在 100 级解锁。韩服 2026 年 3 月 26 日后甚至进一步把觉醒任务删除，改成达到等级自动完成觉醒。这意味着你的角色成长表至少要有 `base class -> advancement -> 1a -> 2a -> neo/self-awakening` 这五个阶段，并支持“剧情触发式”和“自动等级触发式”两种开关，因为不同服务分支在 2026 年已经不一致。citeturn42view2turn42view3

国服 2025 的官方“角色技能优化”给了你一批非常硬的数值级规则：**TP 被删除**；一觉技能原有 3/6/9 级附带效果被删除；三觉后的“SP 消耗量减少系统”被删除；一觉被动 SP 从 45 调到 30，二觉被动 SP 从 80 调到 45，二觉主动从 `200/100` 改为 `0` 并变为自动习得。对于你要做的一比一服务端，这些不是“平衡改动”，而是**数据模型字段变化**：`tp_tree`、`awakening_level_bonus`、`post-neo_sp_discount`、`auto_learn_flag` 都在不同版本有无切换。citeturn16view0

关于你点名要的**护石 / 符文 / 技能变体**，现在必须按分支理解。旧体系里，全球服官方写得很明确：每个 Talisman 可装 3 个 Rune，同一护石上不能插同种同技能 Rune，Rune 本质是“进阶技能强化媒介”。但在新体系里，全球服 VP 系统已经把这套替换掉：每个可定制技能有 2 个变体选项，角色在 Neo Awakening 后直接获得 5 个 VP 点，可最多定制 5 个技能；另有独立的 Skill Enhancement 页，最多选 3 个技能做伤害/冷却调优，而且这 3 个不再额外消耗点数。中文服 2025-06-12 之后也已下线原护石/符文，改为“技能定制/技能突破”。因此如果你目标是“老 DNF”，就做 `talisman + 3 rune slot`；如果目标是“新 DNF”，就做 `vp_variant[5] + enhancement[3]`。这两者不是同一种系统。citeturn16view4turn18view0turn18view1turn16view1turn41search2

**Buff 换装 / Buff Enhancement** 公开资料也足够落地。全球服官方说明这个系统在 70 级自动解锁，可在背包中选择一个 Buff 技能并登记强化它的装备；韩服指南则直接给出了“最大化 Buff”的组合逻辑：Buff 强化栏可以登记深渊鳞片装备、Buff 称号（最高 +3 级）、稀有上衣（+1 级）+ Buff 白金徽章、稀有下装 + Buff 白金徽章，以及带 Buff 技能词条的宠物。换句话说，Buff 换装不是普通装备切换，而是一个**独立的影子装备页**：它有单独技能选中、独立装载集合、与背包/金库联通、并在结算时把特定来源合并进主 Buff 技能。citeturn16view3turn16view2

## 装备强化、构筑注入与战前准备

强化线的真值可以直接照表落。韩服官方指南写明：强化由 NPC 进行，武器到 `+12`、其他装备到 `+10` 前不会被破坏；增幅需要“异界气息净化后”的维度属性，且与强化互斥；锻造（Refine）只作用于武器独立攻击，最高到 `8`，失败只消耗材料不掉级；继承可以把强化、锻造、增幅、附魔从一件不可交易的稀有及以上装备转移到另一件装备上。这里面最关键的不是 UI，而是四条独立升级轨：`reinforce_track`、`amplify_track`、`refine_track`、`enchant_track`，以及一个 `inherit_payload` 容器。citeturn16view5turn16view6turn16view7turn16view8

**装备耐久 / 修理** 也不是边角料，而是真实参与战斗成本的系统。官方“解体与修理”指南明确说，地下城入口的简易整备机可用于修理，装备耐久下降后可在此恢复；另一个官方页面又说明耐久为 0 时甚至无法使用解体机。官方职业改动公告还提到，某些技能进化选项会按比例改变 `MP 消耗量 / 武器耐久消耗量 / 无色消耗量`。公开社区长文则一致描述：装备在地下城里使用会持续减少耐久，耐久为 0 时不会消失，但会被当作未装备对待，并可做紧急修理。这里能下的工程结论是：耐久至少有**展示值、修理入口、紧急修理、与技能/命中行为挂钩的消耗器**，但“每 Hits / 每 Skill / 每 Cube 的精确耗损公式”并未公开。citeturn37view1turn36search4turn39search8turn39search0

**时装 / 徽章 / 光环 / 宠物 / 称号** 在新版本绝对不是纯外观。官方 Avatar 指南写得很直接：Avatar 自带有用属性，并且有 3/5/8 件套效果；Clone Avatar 会叠加额外属性。韩服徽章指南还把不同部位的徽章孔位和颜色写死了：头/帽红孔，脸/胸黄孔，上/下装绿孔，腰/鞋蓝孔，皮肤/武器/光环多彩孔，稀有上/下装还能打白金徽章。更关键的是，当前官方礼包页已经把这些“外观槽”的战斗注入写得非常明确：光环可给被动技能 +1、元素强化、攻击/Buff 增幅、暴击、命中，并带 2 多彩 + 1 白金孔；称号可给 1–95 技能 +1、50 级主动 +2、命中 +3%；宠物不仅给战斗属性，还能挂一个 V 键基础技能，例如清除异常状态。你的数据模型里，这些槽位必须是**独立装备位**，而不是角色外观 JSON。citeturn26view0turn29search16turn26view1turn27view2turn27view3turn26view5

装备构筑层现在由**Set Point + Tuning + Armory + Conversion + Simulator** 串起来。官方 Set Point 规则是：每套装备有 12 个部位；同套装备的专属点数相加，公用点数只补到“当前专属点数最高的套装”；到达不同点数档位时，名望、最终伤害、Buff 力和套装专属效果逐步解锁。Tuning 会直接提升武器的最终伤害/Buff 力/名望，或提升防具首饰特殊装备的 Set Point；Armory 允许每个角色把 115 装备注册进去，同一装备最多登记 9 份；Equipment Conversion 则可把当前穿着的 115 装备转换成武库里已登记的另一件 115 装。再加上官方 Equipment Simulator 会在模拟穿戴时自动继承旧装备上的强化、增幅、锻造、附魔和融合石，你就能看出当前版本的构筑核心已经不是“换装试伤害”，而是**数据库驱动的整套 Build 状态切换**。citeturn33view6turn33view7turn33view5turn33view8turn33view4

## 组队、副本与攻坚编排

你要的**奶系 / Buffer / Party Buff** 在公开资料里已经能还原到“系统层”，不仅仅是职业定位。官方 Buffer Guide 明确把 Male Crusader / Female Crusader / Enchantress 定义为专门强化他人的职业；官方副本改动又写明，在某些内容里，**Buff 数值会按队伍人数增加，而怪物 HP 仍按原来的 1 人 / 4 人桶位规则计算**。另外，像 Nabel 这类内容还明确写出“频道/地域 Buff 提升的力量与智力不影响 Buffer 的 Buff 技能”，说明副本区域 Buff 与职业 Buff 在结算链上是分开的。citeturn23view1turn15search2turn24view2

**队伍状态同步与支援目标反馈**，官方虽然没有公开网络协议，但已经公开了大量前端反馈规则：对队友生效的恢复技能会同时在目标的 HP/MP 条和队伍头像上出现恢复特效；Buffer 的复活技能只要还有次数，队伍框下方就会显示复活图标；Buff 图标还支持“合并显示”，方便团队查看关键 Buff 剩余时间。这些说明一比一复刻时，队伍系统至少需要同步**Buff 生效对象、恢复/复活可用状态、队伍框图标状态、辅助技能反馈**，否则 UI 和机制联动会完全不像原作。citeturn23view0turn20search7

**Raid 指挥 / Ping / 战场状态板** 的资料已经很成熟。全球服和韩服都公开了 Prey-Isys 与后续 Raid 的 Ping 系统：只有团长能在状态板上给组别打标；状态板会高亮团长选择的 Ping 点；接近 Ping 点时会出现导引标识并伴随音效；在新 Raid 中，同色 Pin 还可同时放置多个并按顺序编号。更进一步，2026 年新 Raid 还新增了**HP Ratio**：创建团本时可在 `0%–500%` 间以 `50%` 为步长调节，全副本所有怪物 HP 都按该比例追加。你的 Raid UI 和服务端规则必须把 `raid_ping`, `status_board_highlight`, `pin_sequence`, `guide_sound`, `monster_hp_ratio` 视作正式协议字段。citeturn16view10turn21view9turn21view8turn15search8

**副本特殊限制规则**是 DNF/DFO 一比一味道最重的部分。官方旧 Raid 与新团本都反复出现同一类规则件：内容总限时、分阶段限时、每战斗限时、团/队共享 coin、相位切换自动复活、回复类消耗品按“每场战斗 8 次”限制、人偶类道具禁用、Hold 免疫、公会/区域 Buff、后退后的恢复惩罚、可重建/重编队投票、失败后周入场次数是否恢复等。比如 Nabel 明确：普通/困难共享 5 个队伍 coin，死亡消耗 1 个，部分 Raid Buff 或特定怪物死亡会补 coin；单人/匹配不使用 coin、而是自动复活；队伍后退会进入“恢复中”惩罚，恢复状态下任一人未结束都不能回战场；失败或放弃后还可发起“攻击队再生成”投票；部分 Boss 战后 HP 会保留到下一次接战。你如果不把这些做成**可配置 RuleSet**，而是硬编码成 dungeon script，后面会完全不可维护。citeturn21view0turn21view1turn21view2turn24view2turn24view3turn24view4turn36search9turn35search4

**伤害归属 / 击杀归属 / 协助归属** 的公开真值目前不是“最后一击”，而更像“贡献阈值系统”。Nabel 官方规则写得很清楚：达到最低贡献度阈值才能拿奖励；贡献度按个人记录，而不是整队统一；Boss 击杀和某些超时事件都会为个人贡献度加分。这个设计对复刻的意义非常大：至少在奖励发放层，DNF/DFO 当前已经把“击杀归属”和“奖励归属”解耦。你可以把 `kill_credit`、`damage_credit`、`reward_contribution` 拆成三套独立字段，不要混成一个。citeturn23view3

## 公开资料还原不到真值的黑箱层

下面这些模块，公开官方资料目前**不足以直接给出一比一真值**：命中/回避/Miss 公式，随机数与伪随机池，同 Tick 的优先级与冲突裁决，技能帧级读条打断窗口，施法保护的精确帧段，目标标签/阵营/可命中对象完整掩码表，连段中的伤害归属与协助归属算法，音频反馈优先级，特效挂点与遮挡/透明度自动裁剪，服务端战斗审计与客服复盘格式，战斗反作弊与非法状态检测规则。公开官方 Open API 的范围只到角色/道具/职业等外显服务数据；公开官方指南也只写到玩家可见规则，而不公开这些引擎内部实现。citeturn40view5turn33view0turn30view0

能作为“侧面证据”的公开材料有两类。第一类是公开工具链，说明静态资源与脚本容器确实可被 clean-room 化提取；第二类是官方专利/后台思路，说明其安全与风控明显偏向**服务器记录、黑名单、处罚点数、统一后台配置**。例如公开专利描述了服务器记录拍卖中标/再挂时间、做阈值判断并对异常账户加处罚点和黑名单；公开后台项目也把版本管理、PVF 管理和客户端辅助配置同步做成后台模块。它们都不足以证明“战斗反作弊怎么做”，但足以说明：如果你想做一比一，**一切关键战斗判定都必须 authoritative 在服务器**，而不是交给客户端。citeturn14search1turn40view1

## 可直接交给开发的 clean-room 复刻路线

你现在最应该冻结的研发对象，不是“整款游戏”，而是**一个 patch-locked combat slice**。我建议你把服务端 schema 至少拆成下面这些主表，并按“官方行为优先、公开容器抽表、黑箱实测补齐”的顺序推进：

- `CharacterGrowth`：职业、转职、觉醒阶段、等级阈值、自动觉醒开关、SP 总量曲线、旧 TP/新 VP 分支开关。  
- `SkillLearn`：技能学习等级、SP 消耗、自动习得、命令串、快捷栏默认位、命令锁、冷却提示、链槽可注册性。  
- `SkillCombat`：伤害段、Cube 消耗、MP 消耗、耐久消耗、无敌帧、霸体帧、抓取分类、Counter 标记、Neutralize / Suppression 系数。  
- `SkillVariant`：旧护石/符文模型或新 VP 变体模型，二者不要混在一张表里。  
- `BuffLoadout`：Buff 强化页、登记技能、登记装备位、称号/上衣/下装/宠物/白金徽章来源。  
- `EquipmentAugment`：强化、增幅、锻造、附魔、继承、调律、升品。  
- `BuildState`：Set Point、Armory 注册、转换目标、模拟器结果缓存。  
- `DungeonRuleSet`：限时、共享 coin、消耗品限制、自动复活、重入次数、HP 比例、地区 Buff、技能削弱/禁用列表。  
- `RaidState`：状态板、Ping、序号 Pin、队伍共享 Buff、贡献度、Boss HP 保留、重建投票。  
- `CombatAuditLog`：事件种子、输入序列、快照哈希、奖励结算依据、异常状态检测结果。

这组表设计不是照抄任何泄露资料，而是从官方公开行为反推出来的最小完整形。它之所以必要，是因为官方已经把技能定制、Buff 强化、Build 切换、Raid 状态板、队伍贡献、装备模拟器都做成了**显式系统**，而不是临时脚本。citeturn18view0turn16view2turn33view8turn33view4turn21view9turn23view3

真正的落地顺序，我建议是这样。先选版本，例如“国服 2025-06-12 后技能定制版”或“韩服 2026-04-23 制压版”；然后用公开 NPK/PVF 工具做静态抽表，再用官方 Open API 校验职业、装备、词条、角色结果值；随后做训练房与 Boss 黑箱采样，把命中、回避、耐久、同 Tick 冲突、归属、伪随机这几块用录像和批量样本拟合出来；最后再做 authoritative server、回放日志、状态审计。不要把公开私服后台当成实现母本，只把它们当“命名线索”和“容器证据”。citeturn40view0turn40view3turn40view1turn40view5

如果你的目标真的是“一比一”，那现在最值钱的，不是再搜更多“攻略”，而是先把**目标版本冻结**，然后按上面的 schema 去做第一轮抽表。我这次查完公开资料后的判断是：**玩家可见层已经足够你定义 60%–70% 的真值；剩下 30%–40% 是必须靠 clean-room 实测补齐的黑箱层。**这也是你能做出“像 DNF”与“就是那一版 DNF”的分水岭。
