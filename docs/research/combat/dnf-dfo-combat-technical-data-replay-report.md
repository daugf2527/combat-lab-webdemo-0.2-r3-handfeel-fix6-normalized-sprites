# DFO/地下城与勇士 技术资料深度研究
> **Status: [TECHNICAL] — 战斗事件记录与确定性回放研究**

## Berserker 技能帧与判定
目前公开资料很少直接记录技能的帧数据。仅有零散信息：例如2019年韩服补丁修复了低帧率下「压制射击」（抛射类、多段技能）丢失命中次数的问题【75†L379-L382】，这暗示狂战士类似Raging Fury的连击判定也以帧为单位进行触发。**Raging Fury**通常被认为是“0帧”释放、多段伤害的技能（10次以上击中），对应的FrameDataAction字段可设为：`startupFrames≈0`，`activeFrames`很短（每击约1帧），`recoveryFrames`很低，可能无额外霸体或硬直。其他狂战士核心技（如**血之狂乱**、**狂暴**）暂无公开帧表，仅能假定其多段击中、伤害衰减等由动画数据定义。
**快速起身（Quick Rebound）**效果可参考官方维基：施放后即时获得较长的无敌时间及短暂霸体【51†L35-L43】【51†L68-L72】。例如Lv1时提供3.0秒无敌和0.30秒霸体【51†L68-L72】。在FrameDataAction中对应字段可为：`hasInvulnerability=true`, `invulnDuration≈3000ms`，`hasSuperArmor=true`, `superArmorDuration≈300ms`。映射示例：`startupFrames=0`（瞬发），`activeFrames`涵盖整段下蹲无敌时间，`recoveryFrames`极短，`hitStop`视无敌结束时的动画是否硬直而定。**技能状态**方面，狂战「怒气爆发」曾出现“受击击飞力异常”漏洞【32†L1-L4】（官方修正后未知具体改动），**血之狂暴**等被动提升攻击，这类属性在数据表中对应`critBoost`/`atkMultiplier`字段。需要注意，所有判定数据最终需通过解析.ani/.pvf文件获得；现有资料无法给出完整数值，只能作字段占位和逻辑映射。

## ANI/PVF/NPK/IMG 格式与解析
DFO客户端资源文件多为**.NPK**容器（存图集）和**.PVF**脚本/数据以及**.ANI**动画文件。开源工具链可参考：LHCGreg的**DFOToolBox**提供读取.NPK的库和查看工具【72†L299-L307】；UlyssesWu的**Oniki**是另一个开源的NPK提取器【73†L220-L228】。对于PVF脚本，可用flwmxd的**DNF-Porting (PVFTools)** 解包【18†L251-L259】。Ani动画方面，社区工具“梦太晓Ani编辑器”可视化编辑.ani帧序列（支持添加霸体等标签）【46†L125-L133】。
**干净室模式建议：** 定义纯净的技能数据架构（schema），例如将.ani/.pvf等提取内容映射到`FrameDataAction`字段：动画文件名(`aniFile`)、每帧持续(`frameDelays`)、帧偏移和碰撞箱(`frameOffsets`,`hitboxes`)、帧标记(`frameTags`如霸体、无敌、硬直等)、技能效果触发时间点(`effectTriggers`)、基础伤害和属性系数等。解析器设计可参考：先用NPK解包到IMG（图像序列）、再用ANI工具读取帧列表和标记。**测试夹具（fixtures）**可包含典型.ani/.pvf数据和对应解析结果的JSON示例，方便单元测试。法律合规方面，仅解析文件格式和公布自己生成的空结构体/schema不会侵犯著作权，但**禁止分发实际游戏资源**。可使用开源库（Apache2许可）【72†L299-L307】【73†L220-L228】作为参考实现而不直接引用非公开代码，以规避版权问题。

## 伤害公式与异常状态
DFO中**伤害类型**分为百分比伤害和固定伤害。玩家社区总结的经典模型：物理/魔法攻击力（装备提供，不随角色自然成长）经过角色属性（STR/INT）增幅后乘以技能百分比，再加上固定伤害部分【58†L124-L132】【58†L139-L147】。例如，粗略公式可表示为`总伤害 ≈ (武器攻+强击值)*(1+STR/250)*技能系数 + (固定伤害值)`【58†L124-L132】【58†L139-L147】。**护甲穿透**（Piercing）由强化提升，忽略防御（终极敌人防御可达85%）【58†L124-L132】。固定伤害基于“独立攻击力”(IA)【58†L139-L147】，强化附加IA，固定伤害技能对武器选择不敏感【58†L139-L147】。
**异常伤害**计算通常是定时伤害（DOT）：如出血、灼烧、感电、中毒。根据资料，出血伤害在3秒内每0.5秒触发一次（共6次伤害）【60†L54-L58】；感电为10秒内每1秒一次（共10次），且首次10连击触发80%量伤害，余20%分解后续触发【60†L64-L68】；灼烧为5秒内每0.5秒一次（10次）【60†L75-L79】；中毒为5秒内每0.5秒一次（10次）【60†L88-L93】。出血伤害随层数增加线性叠加（每层+1%，11层满层+10%【60†L54-L58】）。对应游戏字段可为：`statusDamageDuration`、`tickInterval`、`stackMultiplier`等。**属性伤害加成**（火焰、冰冻等）通过词条/技能增强，按百分比直接加成最终伤害（资料暂无具体公式源，仅游戏内可查询）。**无敌/霸体**技能如Quick Rebound (前述)在计算时忽略所有外来伤害判定；**硬直/击退**效果则在角色状态中处理，不直接影响伤害公式。所有数据都需结合客户端数据读取，以上公式和数值来源于社区总结【58†L124-L132】【60†L54-L58】。

## 怪物AI与行为模式
DFO怪物普遍由**状态机和触发器**控制：它们具有各种保护状态（超级护甲、无法击飞、无法抓取等）【80†L38-L41】【80†L46-L48】。例如，“无法抓取”状态使怪物对普通抓取技能免疫，只能被**超级抓取**控制【80†L38-L41】；“控制免疫”状态相当于同时附加了霸体、无法抓取和建筑免疫标签【80†L46-L48】。**怪物状态**还包括重量（影响被击高度）、起身速度（倒地后回复）、僵直抗性、反击技能内置冷却等【80†L13-L21】【80†L46-L48】。BOSS行为通常分阶段：在特定血量或计时触发状态切换和技能序列。虽然官方未公开AI脚本，但资深玩家会通过爆肝或反编译**脚本文件**来推导BOSS模式。正如一则论坛讨论所述：“老玩家能够阅读脚本来了解BOSS的模式”，新BOSS上线前往往已有攻略【78†L6-L10】。实践中，可将怪物AI抽象为每个技能节点的**冷却和随机权重表**（Aggro/select率），配合动画脚本。在manifest中建议字段：`aiStateList`（动作列表）、`aggroRange`、`phaseTriggers`（血量阈值）、`stateTags`（不可抓取、霸体等）、以及每个动作的`cooldown`、`hitstunFrames`、`knockdownDelay`等。

## 确定性战斗回放系统
为实现可重现的战斗回放，应采用**固定时间步（fixed timestep）**的模拟循环，并记录全部玩家输入及随机种子。通用做法是：游戏开始时记录初始随机数种子，运行时每帧存下玩家输入及帧号【68†L220-L228】（或与固定帧速同步的时间戳）。回放时重置状态和随机种子，按照记录的输入序列重演游戏【68†L220-L228】。如StackOverflow提到：“要创建回放，需要存储用户输入和随机种子，然后重新播放相同输入”【68†L220-L228】。关键是保持完全**确定性**：同一输入序列必然产生相同结果【70†L151-L159】，因此模拟中禁止使用无序异步调用或不受控随机（遇到随机事件时需使用事先记录的种子）。一种流行的网络模型是GGPO式的输入延迟+预测+回滚，虽然这用于网络延迟，但也验证了固定步、输入日志的思路【70†L110-L119】【70†L133-L140】。对于调试，可定期比较状态哈希或差异，以确保版本兼容。总之，一个连贯的链条为：**输入队列(Input) → 判定命中(HitDecision) → 伤害结算(Damage) → 反应(Reaction) → 回放录制(Replay)**，其中每环节的数据字段（如`inputFrame`,`damageValue`,`hitState`等）都应记录，以便完整重现战斗。
